Standing on other people's work

This writeup exists because others published first. Where my numbers disagree with theirs, it's hardware or model context — not a claim that they're wrong. Stacks compare only against themselves.

  • unsloth — NVFP4 checkpoints, CUTE_DSL_ARCH=sm_121a, b12x probe, published MMLU-Pro
  • r0b0tlab — the key phrase: "B12X routed experts + MTP k=2, Triton draft MoE"
  • walterra.dev — persistent vllm-cache mount, fork-starvation warning at high util
  • eugr/spark-vllm-docker — the image the unsloth stack runs on; drop_caches; honest "not fully optimized" note
  • AEON-7 — uncensored images, DFlash drafters, Spark-measured model cards

The daily-driver recipe

What one Spark delivers with the unsloth stack on eugr/spark-vllm:latest:

  • 35B-A3B: 89–95 tok/s single-stream · 10.65× full-262k KV at U=0.55 (21.46× at 0.65) · 926 tok/s aggregate at 64 streams
  • 27B dense: 33.5–35.5 tok/s with an AEON DFlash drafter grafted onto unsloth weights (+35%)

Three commands. No custom builds. Leave --moe-backend auto — I measured the b12x path as a loss on this box.

docker pull eugr/spark-vllm:latest
mkdir -p ~/vllm-cache

docker run -d --name unsloth35b --gpus all --ipc=host --net=host \
  -e CUTE_DSL_ARCH=sm_121a \
  -v ~/.cache/huggingface:/root/.cache/huggingface \
  -v ~/vllm-cache:/root/.cache/vllm \
  --entrypoint vllm eugr/spark-vllm:latest \
  serve unsloth/Qwen3.6-35B-A3B-NVFP4-Fast \
  --served-model-name unsloth35b \
  --host 0.0.0.0 --port 8888 --trust-remote-code \
  --kv-cache-dtype fp8 --moe-backend auto \
  --gpu-memory-utilization 0.55 --max-model-len 262144 \
  --max-num-seqs 8 --max-num-batched-tokens 32768 \
  --enable-chunked-prefill --async-scheduling --enable-prefix-caching \
  --speculative-config '{"method":"mtp","num_speculative_tokens":3}' \
  --reasoning-parser qwen3 --tool-call-parser qwen3_xml --enable-auto-tool-choice

⚠️ Universal rules: never set util above 0.80 (OS lives in the same 121 GiB). Between switches: stop → drop_caches → boot. After every boot, send one real completion — /health lies. Full recipes for both stacks: DEPLOY.md.

b12x + MTP: the recipe nobody documents — then the plot twist

unsloth's Spark advice (--moe-backend flashinfer_b12x) dies the moment you add MTP: 

ValueError: moe_backend='flashinfer_b12x' is not supported for unquantized MoE.
Expected one of ['triton', 'flashinfer_trtllm', 'flashinfer_cutlass', 'aiter'].

The MTP draft head ships unquantized, and --moe-backend is global. The fix: SpeculativeConfig has its own moe_backend field. h/t r0b0tlab — his config implied this all along. 

export CUTE_DSL_ARCH=sm_121a
vllm serve unsloth/Qwen3.6-35B-A3B-NVFP4-Fast \
  --moe-backend flashinfer_b12x \
  --speculative-config '{"method":"mtp","num_speculative_tokens":2,"moe_backend":"triton"}'

Log confirms it: Using 'FLASHINFER_B12X' NvFp4 MoE backend. Then I measured it.

Prefill flat · decode −11% · KV pool cut to a third. The 35B-A3B is a GDN hybrid — prefill runs on a Triton/FLA kernel and never touches the MoE GEMMs b12x accelerates. Decode on GB10 is bandwidth-bound at ~273 GB/s. unsloth's 1.79× is real where MoE compute is the bottleneck. On this box, it isn't.

One honesty check: unsloth warn that without b12x you fall back to Marlin W4A16. I never hit that — auto picks Cutlass, not Marlin. Their advice is right for the case they describe; it didn't apply here.

What gpu-memory-utilization actually buys

On unified memory: KV pool, not speed.

  • 35B: 0.55 → 10.65× of 262k · 0.65 → 21.46× — decode identical
  • 27B: 0.55 → 1.9× · 0.74 → 4.24× — decode identical
Hard ceiling is real: spec-decode buffers scale with max-num-seqs outside util accounting. U=0.80 + 4 DFlash seqs hard-locked my box. Power button. See TROUBLESHOOTING.md.

Speculative decoding is a category property

Same 27B (unsloth weights + AEON DFlash drafter graft), same config, non-streaming, 200-token outputs — only the prompt type changed:

Category k=10 k=15 (AEON's headline k)
math 44.6 33.5 (spread 21.8–43.3)
JSON 36.7 41.6
coding 32.6 35.7
prose 27.3 27.6
KV @262k 4.24× 3.77×
When you see a "peak" tok/s for a spec-decode model, ask which category it was. Optimal draft length k belongs to the checkpoint+drafter pair — k=15 is AEON's tuned headline; on my grafted drafter it's net-flat with worse variance and a smaller KV pool. k=10 stays. The "gap" with AEON's published 27B numbers dissolves once category and k are understood.

Where a win is claimed, three-rep ranges don't overlap. Ornith's own runs disagreed by 15% — that's why its result is "dead heat," not a 4% claim.

Also settled: AEON checkpoints auto-pick Marlin. Forcing Cutlass rejects at boot (QuantKey(u8, scale(f8e4m3fn…))). Marlin is the only backend for that quant format. No speed left on the table.

Method — and two conclusions I got wrong

  • One variable at a time. Same boot script, same bench, same box. Image A/Bs swap only IMAGE=.
  • Zombie rule. /health is a different process from the engine. Every number here comes from an engine that answered a real completion first.
  • Reps until the spread is smaller than the claim. Two reps of ornith disagreed by 15% — that can't resolve a 4% delta.
  • Prefill ≠ decode. A decode-only bench cannot tell you whether a kernel change helped prefill.
My bench streams tokens and reads ~15–25% lower than non-streaming tooling. Ornith on one unchanged image: 68.6–71.9 here vs 92.7 non-streaming. Do not compare these numbers against other people's — only same-script numbers are valid.

Wrong conclusion #1 — "new image is 20% slower"

Compared a fresh streaming measurement to a stored non-streaming baseline. Same-script control: the images were identical. The 20% was measurement method.

Wrong conclusion #2 — "b12x is dead on sm_121"

Two flag failures said so. The probe said b12x gemm True | b12x moe True all along. Real blocker was the unquantized MTP draft head — fixed by SpeculativeConfig.moe_backend.

Nearly wrong #3

Almost kept the old AEON image off a one-model (ornith) test. That would have cost the +22% on the 27B.

Five ways the box dies

All specific to unified memory. Full catalog in TROUBLESHOOTING.md.

  1. Ghost memory — engine gone, RAM not. Fix: sync; echo 3 | sudo tee /proc/sys/vm/drop_caches
  2. docker restart never works — re-enters ghost memory. Sequence: stop → drop_caches → start → real request
  3. Zombie boots — health OK, first token dies. Always send a real completion
  4. DFlash buffers invisible to util — U=0.80 + N=4 hard-locked the machine
  5. KV per seq varies 5× — don't reuse util across models (AEON 27B needs 27.14 GiB per 262k seq)
